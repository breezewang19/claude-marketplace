# Review Iteration - 审查与迭代修改

## 用途
为 Step 6 提供用户审查、修改处理和迭代循环逻辑。跟踪修改次数和用户确认状态。用户明确确认后才进入验证。每次修改后重新生成流程图。

## 初始审查展示

将生成内容和流程图一起展示，不要分开确认:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 生成的审查点内容
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

{checkpoint_name}

{review_steps}

{document_types}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
【推荐信息点清单】
{inferred_information_points}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📊 审查逻辑可视化

【ASCII 流程图】
{ascii_diagram}

【Mermaid 流程图】
{mermaid_code_block}

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

信息点优先使用 CSV 字段名，无 CSV 数据时从审查步骤推断。按文书类型分组，使用「」括号。

## 审查提示

展示内容后，用一句话引导审查，不要逐条问问题:

> 请查看以上内容，如需修改直接告诉我，没问题就继续验证。

## 响应类型处理

**1. Approval** — "是"/"正确"/"没问题"/"可以"/"确认"
→ 进入 Step 6 验证

**2. Rejection without details** — "否"/"不对"/"有问题"
→ 询问需要修改什么：审查步骤/判断条件/添加场景/修改文书类型

**3. Specific modification** — 用户直接说明修改内容
→ 解析修改目标(步骤/条件/文书)，应用修改，进入迭代循环

**4. Partial approval** — "大部分正确，但是..."
→ 保留已确认部分，聚焦问题，应用针对性修改

**5. Complete regeneration** — "重新生成"/"重来"
→ 询问是否修改基本信息，是则回 Step 2，否则回 Step 4

## 迭代循环

每次修改后:
1. 更新内容，确保必需元素仍在
2. 重新生成 Mermaid + ASCII 流程图
3. 展示更新内容和变更摘要
4. 重复审查问题

展示格式:
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🔄 内容已更新（第 {N} 次修改）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【本次修改】{modification_summary}
【更新后的内容】{updated_content}
【更新后的流程图】{updated_diagrams}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

## 迭代限制

- **3次后**: 温和提醒 — "我们已经进行了 3 次修改。当前内容已经比较完善。你可以：1.确认继续验证 2.继续修改 3.查看修改历史"
- **5次后**: 强烈建议 — "建议先验证当前版本，后续再调整。或考虑重新梳理需求。"
- 限制为建议性，用户坚持则继续

## 修改历史

记录每次修改的用户要求、已应用变更和修改时间。用户请求时可展示历史。

## 示例

**Example 1: 修改判断阈值**
```
User: 判断条件不对，应该是超过24小时才有问题，不是12小时

System: 好的，我将修改判断条件：
- 原条件：超过12小时判定为存在问题
- 新条件：超过24小时判定为存在问题
正在更新内容...
```

**Example 2: 添加遗漏场景**
```
User: 还需要检查传唤证上是否有办案人员签名

System: 好的，我将添加签名检查：
- 在提取传唤时长之前
- 检查传唤证上是否有办案人员签名
- 如果没有签名，判定为存在问题
正在更新内容...
```

## 辅助提示

用户难以表达修改时:
- "不太对"无细节 → 列出可修改的部分供选择
- 模糊修改请求 → 确认具体改什么、在哪一步、改后预期效果
- 复杂修改 → 逐步进行，每步确认

## 确认后过渡

用户确认后:
```
✅ 内容已确认！
- 审查点名称：{checkpoint_name}
- 修改次数：{iteration_count}
接下来将进行内容验证...
```
直接进入 Step 6，不额外询问。
