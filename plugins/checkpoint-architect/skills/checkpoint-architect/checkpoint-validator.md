# CheckPoint Validator - 审查点内容验证器

## 用途
验证用户生成的审查点内容，确保符合系统格式要求，避免与Python程序自动生成的元素冲突。

## 输入格式
请提供需要验证的审查点内容（纯文本或markdown格式）。

## 验证规则

### 1. 必需元素检查 (Required Elements)

**必须包含：**
- ✅ 明确的步骤结构（第一步、第二步、第三步...）
- ✅ 文书缺失处理逻辑（若缺少文书，则判定为...）
- ✅ 明确的判定结论（判定为"存在问题" / "不存在问题" / "缺少文书材料"）
- ✅ 完整的审查流程（定位文书 → 提取信息 → 分析判断 → 输出结论）

**验证方法：**
- 检查是否存在"第一步"、"第二步"等步骤标记
- 检查是否包含"若缺少"、"若不缺少"等文书缺失判断
- 检查是否包含"判定为"关键词及三种状态之一

### 2. 格式冲突检查 (Format Conflicts)

**禁止包含以下元素（Python程序会自动添加）：**
- ❌ "审查点 X:" 前缀（X为数字）
- ❌ "审查方法 (提示):" 标题
- ❌ "审查所需文书类型:" 标题

**验证方法：**
- 使用正则表达式检测：`审查点\s*\d+\s*:`
- 检测完整字符串：`审查方法 (提示):`
- 检测完整字符串：`审查所需文书类型:`

**说明：**
Python程序使用以下模板自动组装：
```
审查点 {point_number}: {point_name}
审查方法 (提示):
{review_steps}
审查所需文书类型: {required_documents}
```
用户只需提供 `{review_steps}` 部分的内容。

### 3. 逻辑完整性检查 (Logic Completeness)

**必须覆盖的场景：**
- ✅ 正常情况：文书齐全且符合要求
- ✅ 异常情况：文书存在但不符合要求
- ✅ 缺失情况：所需文书不存在

**验证方法：**
- 检查是否有"若...则..."的条件判断
- 检查是否覆盖文书存在和不存在两种情况
- 检查逻辑分支是否完整（if-else结构）

**常见问题：**
- 只处理了文书存在的情况，未处理缺失情况
- 只有正常流程，缺少异常处理
- 判定标准模糊，没有明确的"存在问题"/"不存在问题"结论

### 4. 语言规范检查 (Language Standards)

**术语规范：**
- ✅ 使用法律专业术语（如"犯罪嫌疑人"、"合规性"、"法定期限"）
- ✅ 文书名称使用书名号《》
- ✅ 表达清晰、逻辑严谨
- ✅ 与现有审查点风格一致

**验证方法：**
- 检查文书名称是否使用《》
- 检查表达是否清晰（避免"可能"、"大概"等模糊词）
- 对比参考示例，确保风格一致

**参考风格示例：**
```
第一步：定位核心文书。从当前涉案人员的文书库中，查找并定位所有与该人员相关的《传唤证》及对应的《讯问笔录》。
第二步：判断传唤性质。
- 若缺少以上文书，则判定为"缺少文书材料"，并具体列明缺失的文书名称。
- 若不缺少，则提取《传唤证》中的关键信息，确认其被书面传唤的事实。
```

## 输出格式

### 验证通过
```
✅ 验证通过

该审查点内容符合所有格式要求：
- [✓] 包含明确的步骤结构
- [✓] 处理了文书缺失情况
- [✓] 有明确的判定结论
- [✓] 未包含冲突格式元素
- [✓] 逻辑完整，覆盖所有场景
- [✓] 语言规范，符合专业标准

该内容可以直接使用。
```

### 验证失败
```
❌ 验证失败

发现以下问题：

【格式冲突】
- 第3行包含"审查点 1:"，这会与程序自动生成的格式冲突
  修复建议：删除"审查点 1:"前缀，直接从步骤内容开始

【必需元素缺失】
- 缺少文书缺失处理逻辑
  修复建议：添加"若缺少以上文书，则判定为'缺少文书材料'"

【逻辑不完整】
- 只处理了文书存在的情况，未处理不符合要求的异常情况
  修复建议：添加异常判断分支，如"若总时长超过24小时，则判定为'存在问题'"

【语言规范】
- 第5行"传唤证"未使用书名号
  修复建议：改为《传唤证》
```

## 使用示例

### 示例1：验证通过的内容
```
输入：
第一步：识别审查对象。
确认当前涉案人员是否为犯罪嫌疑人。如果是，则从该人员的文书库中查找其所有的《传唤证》及对应的《呈请延长传唤报告书》。若缺少以上文书，则判定为"缺少文书材料"，并具体列明缺失的文书名称。
第二步：审查传唤时限。
针对该人员的每一份《传唤证》，执行以下操作：
a. 提取关键时间：从《传唤证》中提取"被传唤人到达时间"和"被传唤人离开时间"。
b. 计算传唤总时长：计算从"被传唤人到达时间"到"被传唤人离开时间"的总时长（以小时为单位）。
c. 判断合规性：
 - 若总时长超过12小时但未超过24小时，则检查是否存在对应的《呈请延长传唤报告书》。
 - 若总时长超过24小时，则直接判定为当前人员"存在问题"，并应指明具体问题。
第三步：得出结论。
- 若该人员的任何一次传唤出现以下情形，则判定为当前人员"存在问题"，并需指明该人员具体传唤证文号及超时时长：
 1. 总时长超过12小时但未超过24小时，且未找到对应的《呈请延长传唤报告书》。
 2. 总时长超过24小时。
- 否则，判定为当前人员"不存在问题"。

输出：
✅ 验证通过
该审查点内容符合所有格式要求，可以直接使用。
```

### 示例2：验证失败的内容
```
输入：
审查点 1: 传唤时长是否合规
审查方法 (提示):
检查传唤证上的时间，看看是否超过12小时。

输出：
❌ 验证失败

发现以下问题：

【格式冲突】
- 第1行包含"审查点 1:"，这会与程序自动生成的格式冲突
- 第2行包含"审查方法 (提示):"，这会与程序自动生成的格式冲突

【必需元素缺失】
- 缺少明确的步骤结构（第一步、第二步...）
- 缺少文书缺失处理逻辑
- 缺少明确的判定结论

【逻辑不完整】
- 未处理超过24小时的情况
- 未处理需要延长传唤的情况
- 未说明如何"检查"和"看看"

【语言规范】
- "传唤证"应使用书名号：《传唤证》
- "看看"表达不够专业，建议改为"审查"、"核查"
```

## 验证流程

当用户提供审查点内容时，请按以下步骤执行验证：

1. **格式冲突检查**：优先检查是否包含禁止元素
2. **必需元素检查**：确认必需元素是否齐全
3. **逻辑完整性检查**：分析逻辑分支是否完整
4. **语言规范检查**：检查术语和表达是否规范
5. **生成验证报告**：按输出格式提供详细反馈

## 注意事项

- 验证器只检查格式和结构，不评判审查逻辑的法律正确性
- 如果用户提供的是完整的prompt文件（包含"审查点 X:"等），应提示用户只需提供审查步骤部分
- 对于边界情况，采用宽松原则：只要不影响程序解析，就视为通过
- 提供的修复建议应具体、可操作，避免模糊指导
