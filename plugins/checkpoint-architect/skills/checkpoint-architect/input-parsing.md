# Input Parsing - 基本信息收集与解析

## 用途
为 Step 2 提供输入解析、文书类型处理和验证逻辑。收集审查点名称和文书类型，两者都收集到后直接进入下一步，无需额外确认。已确认的信息不再重复询问。用户提供新信息时重置确认状态。

## 输入模式识别

**Pattern 1: Complete Input (Name + Types)**
- 格式: `{checkpoint_name} {document_types}`
- 示例: "传唤时长是否合规 传唤证，呈请延长传唤报告书"
- 识别: 名称含疑问词(是否/有无)或以合规/检查/审查结尾；文书类型跟在名称后，含逗号或《》

**Pattern 2: Name Only**
- 含疑问词: 是否、有无、能否
- 以合规、检查、审查、规范性、完整性结尾
- 无文书类型标识(无逗号、《》、"文书"关键词)

**Pattern 3: Types Only**
- 含逗号分隔项、《》括号
- 含文书关键词: 证、书、表、单、函、通知
- 不含疑问词或审查相关术语

**Pattern 4: Natural Language**
- 含意图关键词: 想、需要、检查、审查
- 提取名称: 检查/审查 + {target}
- 提取文书: 需要/涉及 + {documents}
- 注意连词: 和、与、及、以及

**Pattern 5: Multi-Message Input**
- 用户跨多条消息提供信息
- 逐步累积，缺少哪个问哪个

## Follow-Up Question 规则

缺少哪个问哪个，都有了就直接继续，无需额外确认。用户要求修改时只改指定部分。

- 缺名称: "审查点的名称是什么？" + 示例
- 缺文书类型: "需要审查哪些文书？" + 示例
- 都有: 直接进入下一步
- 用户要求修改: 只改指定部分

## 文书类型解析

**分隔符:** 逗号(，)、分号(；)、"和"、"与"、"及"

**格式变体处理:**
- 有《》: "《传唤证》，《呈请延长传唤报告书》"
- 无《》: "传唤证，呈请延长传唤报告书"
- 混合: 均按分隔符拆分后统一处理

**规则:**
1. 按分隔符拆分
2. 去除《》括号
3. 去除首尾空白
4. 内部存储为列表
5. 显示时加《》并用逗号分隔

## 输入验证

**审查点名称:**
- 非空，5-50字符，含有效中文
- 失败提示: "审查点名称似乎不完整，请提供更详细的名称。"

**文书类型:**
- 至少1个，每项非空，3-30字符
- 失败提示: "文书类型格式有误，请用逗号分隔多个文书类型。"

```
请选择生成模式：
A. 快速模式 — 用自然语言描述审查逻辑

```

```
System: ✓ 收到。审查点：传唤时长是否合规，文书：《传唤证》《延长报告书》

请选择生成模式：
A. 快速模式 — 用自然语言描述审查逻辑
```
